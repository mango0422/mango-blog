# Part 1 요약노트 — 데이터 모델링의 이해 (SQLD 1과목 전반부)

생성일: 2025년 11월 12일 오후 10:28

### 🧩 1-1 데이터 모델의 이해

**모델링의 개념**

- 현실 세계의 업무와 데이터를 **추상화·구조화**하여 DB에 표현하는 과정.
- 핵심 목표: 데이터 구조, 관계, 관리 방법을 명확히 정의.

**모델링의 특징**

1. **단순화**: 불필요한 세부 제거, 핵심 요소만 남김.
2. **추상화**: 다양한 현상을 일정한 표기법에 맞춰 표현.
3. **명확화**: 모호함 제거, 의사소통 용이.

**3가지 관점**

- 데이터 관점: 저장·접근·관리 방식 정의.
- 프로세스 관점: 데이터의 흐름과 변환 과정.
- 데이터+프로세스 관점: 시스템 전체 동작 파악.

**유의점 (3대 문제)**

- 중복(Duplication): 같은 정보 중복 저장 금지.
- 비유연성(Inflexibility): 사소한 업무 변화에도 모델 변경 방지.
- 비일관성(Inconsistency): 상반된 정보 발생 방지.

**모델링 3요소**: 엔터티(Entity), 속성(Attribute), 관계(Relationship)

**모델링 3단계**

1. **개념적 모델링**: 핵심 엔터티 도출, ERD 작성.
2. **논리적 모델링**: 속성·식별자 정의, 정규화.
3. **물리적 모델링**: 실제 DB 구조 설계(인덱스, 저장 구조 등).

**스키마 3단계**

- **외부**(사용자 관점) → **개념**(전체 논리) → **내부**(물리 저장 구조)
- **독립성**
  - 논리적: 개념 스키마 변경이 응용에 영향 X
  - 물리적: 내부 스키마 변경이 논리 구조에 영향 X

---

### 🧱 1-2 엔터티(Entity)

**정의**: 현실 세계에서 독립적으로 식별 가능한 객체(사람, 사물, 사건 등).

**특징 요약 (6가지)**

1. **유일한 식별자**로 구분 가능.
2. **업무에 필요한 정보**만 포함.
3. **2개 이상의 인스턴스**로 구성.
4. **속성 보유 필수.** (2개 이상)
5. **업무 프로세스에 사용**되어야 함.
6. **다른 엔터티와 관계 존재.**

**엔터티 분류**

- **형태 기준**: 유형(사원), 개념(상품), 사건(주문)
- **발생 시점 기준**: 기본(고객), 중심(계약), 행위(주문이력)

**명명 규칙**

- 현업 용어 사용, 약어 최소화, 단수명사, 유일 이름, 생성 의미 반영.

---

### 🔢 1-3 속성(Attribute)

**정의**: 엔터티가 가진 고유한 성질. DB에서는 **컬럼(Column)** 으로 표현.

**특징 요약**

- 업무상 필요한 정보만 포함.
- 주식별자에 함수적으로 종속.
- **원자성(Atomicity)**: 하나의 속성은 한 값만 가짐.
- 다중값 속성은 별도 엔터티로 분리.

**함수적 종속성**

- A → B: A가 B를 유일하게 결정.
  - 완전 종속: PK 전체에 종속.
  - 부분 종속: PK 일부에 종속 → 2NF 위배.

**속성 분류 요약**

- **특성 기준**: 기본 / 설계 / 파생 속성
- **역할 기준**: PK / FK / 일반 속성
- **분해 기준**: 단일 / 복합 / 다중값

**도메인(Domain)**

- 속성이 가질 수 있는 값의 범위 (데이터 타입, 크기, 제약 포함)

---

### 🔗 1-4 관계(Relationship)

**정의**: 엔터티 간 논리적 연관성.

**종류**

- 존재적 관계: 한 엔터티의 존재가 다른 엔터티에 의존.
- 행위적 관계: 행위(이벤트)에 의해 발생.

**구성요소**: 관계명, 차수(Cardinality), 선택성(Optionality)

**차수 종류**

- 1:1 (사원–사원카드)
- 1:N (부서–사원)
- N:M (학생–강의 → 수강 엔터티로 분리)

**페어링(Pairing)**: 인스턴스 간 구체적 연관. (예: 학생A–강의B)

---

### 🆔 1-5 식별자(Identifier)

**정의**: 엔터티 인스턴스를 구분하기 위한 고유 속성 (논리모델: 식별자 / 물리모델: Key)

**특징 (4대 조건)**

1. 유일성
2. 최소성
3. 불변성
4. 존재성 (NULL 불가)

**식별자 분류 요약**

| 기준      | 종류                    | 설명                        |
| --------- | ----------------------- | --------------------------- |
| 대표성    | 주식별자 / 보조식별자   | 대표성 여부                 |
| 생성 여부 | 내부식별자 / 외부식별자 | 자체 생성 or 관계 통해 생성 |
| 속성 수   | 단일 / 복합             | 하나 or 여러 속성으로 구성  |
| 대체 여부 | 본질식별자 / 인조식별자 | 업무기반 vs 인위적 생성     |

**주식별자 도출 기준**

- 자주 이용되는 속성 우선.
- 이름 등은 피하고 코드 사용.
- 구성 속성은 최소화.

**관계성 분류**

- **강한 개체**: 독립 존재 가능 (고객)
- **약한 개체**: 부모 엔터티 의존 (계좌)

**식별/비식별 관계**

- 식별: 부모 PK를 자식의 PK로 포함(실선)
- 비식별: 부모 PK를 자식의 일반 FK로 포함(점선)

**Key 종류 정리**

- 기본키(PK), 후보키(CK), 슈퍼키(SK), 대체키(AK), 외래키(FK)

---

### 🧮 1-6 정규화(Normalization)

**개념**: 중복을 제거하고 이상현상(삽입·갱신·삭제)을 방지하는 설계 기법.

**이상현상 종류**

- 삽입이상, 갱신이상, 삭제이상

**정규화 단계 요약**

| 단계 | 조건          | 위배 사례             | 해결 방법   |
| ---- | ------------- | --------------------- | ----------- |
| 1NF  | 원자성        | 컬럼에 여러 값        | 행 분리     |
| 2NF  | 완전 종속     | PK 일부에 종속        | 테이블 분리 |
| 3NF  | 이행 종속     | A→B, B→C              | 테이블 분리 |
| BCNF | 결정자≠후보키 | 비후보키가 결정자     | 테이블 분리 |
| 4NF  | 다중값 종속   | 하나의 컬럼이 여러 값 | 분리        |
| 5NF  | 조인 종속     | 조인 결과 중복        | 분리        |

**반정규화(De-normalization)**

- 성능 향상을 위해 중복 허용, 조인 최소화.
- 조회 속도 ↑, 유지보수·유연성 ↓.

---

✅ **핵심 암기 문장**

- 모델링의 목적은 **현실 세계의 복잡성을 단순·명확하게 구조화**하는 것.
- **엔터티는 업무가 관리하려는 대상**, **속성은 그 대상의 특징**, **관계는 상호연관성**.
- **식별자는 유일성·최소성·불변성·존재성**을 모두 만족해야 함.
- **정규화의 목표는 이상현상 제거 + 중복 최소화**.
